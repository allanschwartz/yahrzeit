PHP Internal Map

Standalone developer documentation generated from the current source headers, functions, constants, includes, and call paths. It records observed behavior only; where the code does not establish an answer, it says unclear.

Overview and architectural boundaries

Project style: this is intentionally a procedural PHP application. Preserve its procedural module boundaries and table-based rendering assumptions; this map does not recommend a class conversion.

Observed control path

cron → bin/yahrzeit_scheduler → bin/yahrzeit → bin/yahrzeit_engine.php → command stream → nc → Arduino V3 controller. The scheduler selects whether a scheduled phase applies; the wrapper handles transport; the engine decides which memorial LEDs should be lit.

  • names.inc.php owns parsing and writing the memorial CSV.
  • panels.inc.php owns static panel geometry.
  • date_support.inc.php owns Gregorian/Hebrew date, sunset, and Shabbat helpers.
  • leds.inc.php owns the mapping from panel IDs/person locations to controller command text. It does not transmit.
  • Screen files primarily render legacy table-based pages and handle their GET/POST flows.

Command-line and scheduler files

Command wrapper and transport

bin/yahrzeit
Kind / purpose
Bash command-line tool. Selects operational mode, invokes the PHP engine as needed, prints dry-run streams, or transports streams to the Arduino controller.
Major functions
usage, set_mode, PHP discovery helpers, run_yahrzeitd, emit_yahrzeit, emit_audit, emit_report, direct emitters, slowcat, transmit_commands.
Constants / global state
Shell variables including CONTROLLER_HOST (default 192.168.13.9), CONTROLLER_PORT (2001), MODE, TRANSMIT, REPORT_KIND, and REPORT_DATE.
Depends on
bin/yahrzeit_engine.php for normal/audit/report paths; PHP with Calendar extension; nc and a timeout command for transmission.
Likely dependents
1viewpanels.php, 6reports.php, bin/yahrzeit_scheduler, cron/manual operators.
Data I/O
Engine modes read project data through PHP modules. --file reads a supplied command file. Wrapper itself has no project data write path.
Controller
Yes by default. --notransmit/--dry-run prints instead. Audit/report modes force non-transmission.
Boundary
Owns transport and direct controller operations; it should not own calendar or Minhag rules.

Unclear areas and human-review TODOs

  • Scheduler installation: source documents sample cron lines but does not show an installed crontab. Deployment state is unclear.
  • Authority/authentication: no access-control mechanism was established from these listed files. Confirm the web-server/authentication boundary before exposing write or manual-control screens.
  • Manual record behavior: 5singlename.php stores onnow, but the current engine’s manual branch lights the record without consulting onnow. Intent is unclear from the inspected code; verify expected operator semantics.
  • Add-record index: 5singlename.php uses yahrzeit_record_cound() + 1. Review whether this intentionally leaves an array index gap, especially before changing the legacy CSV path.
  • Panel control host: 1viewpanels.php contains a documented local-lab host override. Confirm that it is appropriate for the deployment environment before allowing manual wall operation.
  • Schedule settings: Minhag stores on/off timing values, while the scheduler header shows fixed cron examples. The relationship between those settings and deployed cron configuration is not fully established by these files.
